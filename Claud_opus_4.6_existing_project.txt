Claud opus 4.6 (в рамках существующего проекта)


## 1. Обязательные разделы блока памяти

Блок памяти должен быть **многоуровневым** — от самого стабильного (редко меняется) к самому волатильному (меняется каждый чат):

### Уровень 1 — Ядро проекта (почти никогда не меняется)
- **Идентификация проекта**: название, назначение, домен
- **Технологический стек**: языки, фреймворки, ключевые зависимости
- **Архитектура верхнего уровня**: основные компоненты, как они связаны
- **Ключевые соглашения**: стиль кода, паттерны, именование, структура папок

### Уровень 2 — Текущее состояние (обновляется при значимых изменениях)
- **Карта компонентов**: какие файлы за что отвечают, их текущее состояние
- **Принятые архитектурные решения**: что было решено и **почему** (reasoning)
- **Известные ограничения и долги**: что работает не идеально и почему оставлено как есть

### Уровень 3 — Оперативный контекст (меняется каждый чат)
- **Текущая задача**: над чем работаем прямо сейчас
- **Незавершённые действия**: что начато, но не закончено
- **Последние существенные изменения**: что менялось в последних 2–3 сессиях

---

## 2. Что является лишним и раздувает контекст

- **Полная история диалога** — новому ИИ не нужно знать, что пользователь сначала спросил X, потом передумал. Нужен только итог.
- **Промежуточные неудачные попытки** — «пробовали подход A, не сработало из-за B» полезно только если B остаётся актуальным ограничением. Иначе — мусор.
- **Дословные куски кода**, которые существуют в файлах проекта — ИИ может прочитать файлы. Достаточно указать «в файле X реализован компонент Y по паттерну Z».
- **Эмоциональный и социальный контекст** — формулировки вежливости, благодарности, переспрашивания.
- **Дублирование информации** между разделами — одна из главных причин раздувания при многократных переходах.

**Критерий**: если удаление фрагмента не помешает новому ИИ принять правильное техническое решение — фрагмент лишний.

## 3. Наиболее устойчивый формат при многократных переходах

**Структурированный plain text с фиксированными заголовками-маркерами.**

Почему не JSON/YAML:
- При каждом проходе через Gemini Flash возрастает риск синтаксических ошибок
- Жёсткие форматы хрупки — одна сломанная скобка разрушает всё

Почему не свободный текст:
- При многократной «перегонке» свободный текст деградирует: разделы сливаются, приоритеты теряются

**Оптимальный формат — markdown с жёстко фиксированной структурой секций:**

```
## PROJECT_CORE
## ARCHITECTURE  
## DECISIONS_LOG
## CURRENT_STATE
## ACTIVE_TASK
## WARNINGS
```

Маркеры секций должны быть **константными** — Gemini Flash получает инструкцию: «заполни эти секции, не добавляй новых, не переименовывай». Это создаёт **каркас**, который не деградирует при переносах.

Внутри каждой секции — **нумерованные пункты** с однострочными утверждениями. Не абзацы текста, а **факты**:

```
## DECISIONS_LOG
1. Electron без фреймворков (vanilla JS) — для полного контроля над UI
2. Данные хранятся в localStorage + JSON-файл на диск через preload.js
3. Синхронизация отключена — автономный режим по умолчанию
```

## 4. История обсуждений vs. итоговые решения

**Только итоговые решения, но с обязательным сохранением причины (rationale).**

Плохо:
> Решили использовать WebSocket

Хорошо:
> WebSocket вместо polling — потому что нужны real-time обновления таймеров с задержкой <1с

Причина критична: без неё через 5 переходов какой-нибудь ИИ предложит «а давайте заменим WebSocket на polling, будет проще», потому что он не понимает, почему выбор был сделан.

**Исключение**: если обсуждение выявило **неочевидное ограничение**, которое всё ещё актуально, — это нужно сохранить в секции `WARNINGS`:

```
## WARNINGS  
1. Electron не поддерживает SharedArrayBuffer без специальных заголовков — не использовать Web Workers с shared memory
```

## 5. Как избежать деградации контекста при многих переходах

Это **центральная проблема** подобных систем. Решения:

### A. Разделение на «холодную» и «горячую» память
- **Холодная память** (PROJECT_CORE, ARCHITECTURE, DECISIONS_LOG) — **не пересоздаётся** каждый раз. Gemini Flash получает инструкцию: «эти секции перенеси as-is, если пользователь явно не менял их содержимое в текущем чате».
- **Горячая память** (CURRENT_STATE, ACTIVE_TASK) — пересоздаётся заново каждый раз.

Без этого разделения при каждом проходе холодная информация «перефразируется», теряет детали, и через 10 переходов от неё остаётся каша.

### B. Монотонно растущий лог решений
DECISIONS_LOG должен быть **append-only**. Новые решения добавляются, старые не удаляются и не переформулируются. Это предотвращает «эрозию памяти».

Когда лог становится слишком длинным — производится **явная компрессия с участием пользователя**: Gemini Flash предлагает убрать устаревшие пункты, пользователь подтверждает.

### C. Контрольные хэши / маркеры версий
В начало блока памяти добавляется:
```
MEMORY_VERSION: 14
LAST_UPDATED: 2026-06-14
TRANSFER_COUNT: 7
```
Это позволяет отслеживать, сколько раз контекст прошёл через «перегонку», и при достижении порога (например, 10) — предложить пользователю провести полную ревизию.

### D. Якоря — неизменяемые строки
Критически важные факты помечаются маркером `[ANCHOR]`:
```
3. [ANCHOR] Все данные приложений хранятся в apps.json через saveAppsToDisk() 
```
Gemini Flash получает инструкцию: строки с `[ANCHOR]` нельзя переформулировать, удалять или объединять с другими.

## 6. Оптимальный формат ответа Gemini Flash

Gemini Flash должен возвращать **ровно один текстовый блок**, который начинается с метаданных и далее идёт по фиксированному шаблону. Ключевые принципы:

- **Фиксированный порядок секций** — всегда одинаковый, чтобы новый ИИ мог быстро найти нужное
- **Каждый факт — одна строка** — никаких многострочных абзацев
- **Нет вводных фраз** — никаких «Ниже представлен контекст проекта…». Сразу данные
- **Максимальный объём ограничен** — задать лимит (например, 3000–4000 токенов). Заставить Gemini Flash приоритизировать, а не дампить всё подряд

Промпт для Gemini Flash должен содержать:
1. Текущий блок памяти (если есть — предыдущий)
2. Полный лог текущего чата
3. Жёсткую инструкцию с шаблоном
4. Правила: что нельзя удалять (ANCHOR), что нельзя переформулировать (холодная память), что нужно обновить (горячая память)

## 7. Типичные ошибки подобных систем

| Ошибка | Последствие |
|--------|------------|
| **Переформулирование при каждом переносе** | Через 5–7 переходов факты искажаются до неузнаваемости (эффект «испорченного телефона») |
| **Отсутствие приоритизации** | Блок памяти растёт линейно, пока не перестаёт помещаться в контекст нового ИИ |
| **Хранение «как делали» вместо «что решили»** | 80% объёма занято процессом, а не результатом |
| **Отсутствие rationale у решений** | Новый ИИ отменяет обоснованные решения, потому что не знает причин |
| **Одинаковая обработка всех секций** | Стабильная архитектурная информация деградирует так же быстро, как и оперативная |
| **Нет механизма ревизии** | Устаревшие факты накапливаются, противоречат новым, путают ИИ |
| **Дословное копирование кода в контекст** | Код быстро устаревает и занимает непропорционально много места. Лучше описание «что и где» |
| **Нет версионирования блока памяти** | Невозможно понять, когда контекст начал деградировать |

---

**Главный принцип**: блок памяти — это **не пересказ чата**, а **техническая документация проекта**, которая обновляется инкрементально. Чем ближе формат к хорошей проектной документации и чем дальше от «краткого содержания разговора» — тем устойчивее система.
