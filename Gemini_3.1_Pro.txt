Gemini 3.1 pro:

Проектирование системы переноса контекста между LLM — это классическая архитектурная проблема при создании ИИ-ассистентов и оркестраторов. Главный враг здесь — эффект «испорченного телефона» (деградация данных при многократном пересказе) и размытие фокуса.
Ниже представлен разбор архитектуры памяти, которая позволит передавать контекст сложных проектов (например, при разработке десктопных приложений, интеграции WebView, изоляции процессов или создании локальных навигаторов по коду) с минимальными потерями.
1. Обязательные разделы блока памяти
Структура должна быть иерархичной, от общего к частному:
•	System Role / Identity: Краткая установка для нового ИИ (кто он, в какой парадигме работает).
•	Project Vision (Глобальная цель): Что именно разрабатывается. Одно-два предложения, чтобы ИИ понимал общую картину.
•	Architecture & Tech Stack (Стек и архитектура): Явное указание технологий и версий (например, Kotlin, Electron, конкретные паттерны IPC-взаимодействия).
•	Core Guidelines (Правила проекта): Специфические требования к коду, стилистике, обработке ошибок или архитектурным ограничениям.
•	Current State (Текущее состояние): Что уже реализовано на данный момент.
•	Active Task (Активная задача): Точка входа для нового чата. Что конкретно нужно сделать прямо сейчас.
•	File Structure / Context Map (Карта файлов): Список ключевых файлов, затронутых в текущей задаче, и их назначение (без полного листинга кода).
2. Лишние разделы (что увеличивает объем и путает ИИ)
•	Хронология диалога («Пинг-понг»): Пошаговые записи вида «Пользователь попросил Х, ИИ дал код, код выдал ошибку Y, ИИ исправил на Z».
•	Устраненные мелкие баги: Ошибки опечаток, неверных импортов или забытых скобок, которые уже решены и не влияют на архитектуру.
•	Полные листинги кода: Код съедает токены и сбивает ИИ с толку. Код должен быть заменен на сигнатуры (интерфейсы, названия классов и методов с их кратким описанием).
•	Вежливые вводные слова и рассуждения: Любая «вода» от предыдущего ИИ.
3. Формат и структура ответа Gemini Flash
Оптимальный формат — Markdown со строгой структуризацией. Markdown невероятно токен-эффективен, и любая LLM (будь то Gemini, Claude или локальные модели) прекрасно понимает его иерархию.
Я рекомендую использовать гибридный подход: Markdown-заголовки для блоков и списки/ключевые пары для фактов.
Пример идеального вывода Gemini Flash (шаблон):
Markdown
# PROJECT CONTEXT
**Goal:** [Краткое описание проекта, например: Локальный индексатор файлов для ИИ-агентов без отправки всей базы в облако]
**Stack:** [Технологии, например: Python, SQLite, AST-парсинг]

## ARCHITECTURE & RULES
- [Правило 1, например: Использовать только асинхронные вызовы для ввода-вывода]
- [Правило 2: Логика парсинга файлов должна быть изолирована от логики поиска]

## DECISIONS RECORD (ADR)
- **Используем X, а не Y**, потому что [Причина]. (Например: отказались от векторной БД в пользу полнотекстового поиска SQLite из-за требований к легковесности).

## CURRENT STATE
- Реализован модуль X.
- Настроен базовый интерфейс Y.

## ACTIVE CONTEXT (Точка входа)
**Current Task:** [Точная формулировка того, на чем прервался прошлый чат]
**Known Issues:** [Текущие нерешенные баги или архитектурные проблемы]

## RELEVANT FILES
- `core/parser.py`: [описание интерфейса]
- `db/indexer.py`: [описание интерфейса]
4. История обсуждений vs Итоговые решения
Хранить нужно только итоговые решения, но обязательно в связке с контекстом (почему было принято такое решение).
В инженерии это называется Architecture Decision Records (ADR). Если вы просто напишете «Используем паттерн X», новый ИИ может попытаться его оспорить. Если вы напишете: «Мы попытались использовать паттерн Y, но столкнулись с проблемой синхронизации потоков, поэтому перешли на паттерн X» (Отрицательное знание) — новый ИИ учтет этот опыт и не пойдет по ложному пути.
5. Как избежать деградации контекста (Главный вызов)
Если каждый раз просить Gemini: «Сожми этот чат», на третьей итерации контекст превратится в кашу, так как модель будет сжимать уже сжатое (эффект JPEG-сжатия).
Решение: Обновление состояния (State Mutation), а не пересказ.
В Gemini Flash нужно отправлять не просто историю старого чата, а Два блока данных:
1.	Предыдущий блок памяти (Current Memory).
2.	Новые сообщения из последнего чата (New Events).
Промпт для Gemini Flash должен звучать так:
"У тебя есть текущий файл состояния проекта (Current Memory) и транскрипт последней рабочей сессии. Твоя задача — обновить файл состояния. Не меняй разделы Project Vision и Tech Stack, если этого не требовалось. Добавь новые архитектурные решения в Decisions Record. Обнови Current State и полностью перепиши Active Task на основе того, чем закончилась сессия. Выдай только обновленный Markdown-файл."
6. Частые ошибки подобных систем
•	Потеря «Отрицательного знания»: ИИ забывает, какие подходы уже были признаны ошибочными, и в новом чате начинает предлагать их снова.
•	Слишком сильное сжатие: Замена конкретных технических терминов или названий переменных общими словами («реализована функция загрузки» вместо «реализован метод load_graph_data в GraphManager»).
•	Отсутствие четкой задачи для нового ИИ: Новый чат начинается с огромного блока контекста, но ИИ не понимает, что от него хотят прямо сейчас (поэтому раздел Active Task должен быть в самом конце контекстного блока, чтобы ИИ обратил на него максимум внимания).
Чтобы система работала максимально устойчиво, как вы планируете передавать структуру и сигнатуры файлов в блок памяти: ИИ (Gemini Flash) должен сам пытаться извлечь названия файлов из диалога, или вы будете подтягивать реальное дерево проекта из файловой системы перед формированием контекста?
